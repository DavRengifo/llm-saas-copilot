1. System Overview

LLM Copilot est une application web full-stack qui expose trois capacités d'analyse de texte pilotées par un LLM (Large Language Model). Un frontend React communique avec un backend FastAPI qui orchestre les appels vers l'API OpenAI et retourne des réponses JSON typées et validées.

2. Cycle de vie d'une requête

Chaque appel à /analyze/* passe par 6 étapes séquentielles. Une erreur à n'importe quelle étape retourne immédiatement une réponse HTTP appropriée sans aller plus loin.

sequenceDiagram participant C as Browser participant A as Auth Layer participant V as Pydantic participant P as Prompt Builder participant L as LLM Service participant O as OpenAI API C->>A: POST /analyze/feedback + X-API-Key alt Clé manquante A-->>C: 422 Unprocessable else Clé incorrecte A-->>C: 401 Unauthorized else Clé valide A->>V: Valider le body JSON alt Body invalide V-->>C: 422 + détail d'erreur else Body valide V->>P: (text, language) P->>L: SYSTEM_PROMPT + user_message L->>O: chat.completions.create() O-->>L: raw JSON string L->>V: json.loads() + FeedbackResponse(**data) alt JSON invalide V-->>C: 422 LLM returned invalid JSON else Réponse valide V-->>C: 200 FeedbackResponse JSON end end end

Figure 2 — Cycle de vie complet d'une requête /analyze/feedback

Pourquoi temperature=0.3 ? Une température basse rend les sorties du LLM plus déterministes. Pour de l'extraction structurée en JSON, on veut de la cohérence, pas de la créativité. Une temperature=0 serait trop rigide sur des inputs variés ; 0.3 est le bon compromis.

5. Architecture des prompts

Chaque endpoint a son propre module de prompt. Le pattern est identique pour les trois :

flowchart LR subgraph PROMPT["prompts/feedback_analysis.py"] SP["SYSTEM_PROMPT\nPersona + JSON schema strict\n+ règles + exemple"] BUM["build_user_message(text, language)\nInstruction langue\n+ texte brut"] end SP -->|"role: system"| API["OpenAI\nchat.completions"] BUM -->|"role: user"| API API -->|"raw string"| PARSE["json.loads()\n+ Pydantic validation"] PARSE -->|"typed object"| RESP["HTTP Response\n200 JSON"]

Figure 4 — Pattern prompt : system + user → parse → validate → respond

Module promptPersona LLMChamps JSON retournés
feedback_analysis.py Senior CX analyst (10 ans d'expérience) themes · sentiment · priority_issues · executive_summary
document_analysis.py Expert business analyst & technical writer key_points · decisions · actions · summary
extract.py Precise information extraction engine extracted_data → dates / names / amounts
Invariant critique : chaque system prompt contient la consigne "You MUST respond ONLY with a valid JSON object. No markdown, no code blocks, no text before or after." C'est ce qui rend le parsing fiable. Sans cette contrainte, le LLM enveloppe souvent sa réponse dans des balises ```json ... ``` qui font échouer json.loads().

6. Modèle d'authentification

1

Client envoie la requête

Header X-API-Key: <valeur> inclus dans chaque POST vers /analyze/*. Côté frontend, la valeur est lue depuis localStorage (ou VITE_API_KEY env).

2

FastAPI Dependency: verify_api_key

Déclarée au niveau du router via dependencies=[Depends(verify_api_key)] — s'applique automatiquement à toutes les routes /analyze/* sans modifier chaque route individuellement.

✗

Header absent → 422

FastAPI détecte l'absence du header requis avant même d'appeler la dépendance.

✗

Clé incorrecte → 401

La dépendance compare la valeur reçue avec os.getenv("API_SECRET_KEY"). Si différentes → HTTPException(401).

✓

Clé valide → requête traitée

La dépendance retourne la clé, FastAPI passe la main à la route handler.

L'endpoint GET /health est hors du router /analyze et n'est donc pas protégé — c'est intentionnel pour les health checks d'infrastructure.

7. Schémas de données (Pydantic v2)

Pydantic est utilisé deux fois par requête : pour valider l'input du client, et pour valider l'output du LLM. Un seul modèle, deux usages.
